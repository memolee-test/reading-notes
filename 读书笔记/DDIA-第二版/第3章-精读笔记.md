# 第3章 精读笔记：数据模型与查询语言

> 基于 Adler《如何阅读一本书》分析阅读法，按“结构性 -> 诠释性 -> 批判性”整理。

---

## 🔍 检视报告

**归类：** 实用-原理类（Practical-Principle）

**一句话说清大意：** 这章想说明：数据库最重要的选择不是“用什么牌子”，而是“用什么数据模型来表达问题”，因为关系、文档、图、事件流和数据框分别擅长不同的关系形态、查询方式和演化节奏。

**作者要解决的核心问题：**
1. 关系模型和文档模型各自适合什么数据形状？
2. 规范化和反规范化怎么权衡？
3. 当关系从树变成网时，为什么图模型更自然？
4. 为什么某些场景需要事件溯源 / CQRS 这种“写和读分离”的思路？
5. 数据框、矩阵和数组为什么会成为分析与机器学习的主力？

---

## 🔷 结构性阅读

### R1 — 分类

实用-原理类（Practical-Principle）。

- **实用**：直接指导系统设计选型
- **原理**：不是某个数据库的操作手册，而是在建立“模型如何塑造系统”的思维框架

### R2 — X-ray（全章主旨）

> 没有一种数据模型能普适所有问题；你要根据数据之间的关系形态、更新频率、查询方式和未来演化方式，在关系、文档、图、事件流与分析模型之间做选择。

### R3 — 大纲（重构论证骨架）

**第一部分：关系模型 vs 文档模型**
- 数据模型的层次结构：现实世界 -> 应用对象 -> 通用存储模型 -> 字节表示
- 对象关系不匹配与 ORM 的局限
- 一对多 / 多对多关系下，文档切碎为多表的代价
- 规范化 vs 反规范化：写入、读取、一致性、存储空间的权衡

**第二部分：何时使用哪种模型**
- 文档模型适合树状、一次整棵读取的数据
- 关系模型更适合多对一、多对多和稳定模式
- 读时模式 vs 写时模式
- 局部性、查询语言与数据库融合

**第三部分：图数据模型**
- 图适合“网状关系”和可变长度遍历
- 属性图：顶点、边、标签、属性
- Cypher / SQL 递归 CTE / SPARQL / Datalog 的差异
- GraphQL 不是图数据库语言，而是面向前端的查询与响应结构语言

**第四部分：事件溯源与 CQRS**
- 写入以事件日志为真相源
- 读优化视图从事件流派生
- 适合审计、重放、演化和多视图需求

**第五部分：数据框、矩阵与数组**
- 面向分析、统计和机器学习的数据形态
- 关系表可以转成矩阵或稀疏数组
- 这类模型更适合批量运算而不是 OLTP

## 🔷 诠释性阅读

### R5 — 关键字与精确词义

#### 关系 / 文档

| 术语 | Kleppmann 的精确含义 |
|---|---|
| **规范化** | 人类可读信息只存一份，其余位置用 ID 引用 |
| **反规范化** | 信息复制到多处，换取读取方便 |
| **连接** | 用 ID 把被分开的实体拼回一条记录 |
| **切碎（shredding）** | 把一个文档拆进多个关系表 |
| **读时模式** | 数据库不强制结构，读取时由应用解释 |
| **写时模式** | 数据库在写入时强制结构一致 |

#### 图模型

| 术语 | 精确含义 |
|---|---|
| **顶点 / 边** | 图中的实体与关系 |
| **属性图** | 顶点和边都可以带标签、属性和方向 |
| **邻接表** | 顶点记录直接邻居，适合遍历 |
| **邻接矩阵** | 二维数组，适合机器学习与稠密图 |
| **可变长度遍历** | 不知道要走几跳，直到匹配到目标为止 |

#### 事件 / 分析

| 术语 | 精确含义 |
|---|---|
| **事件溯源** | 用不可变事件日志作为真相源 |
| **CQRS** | 写模型和读模型分离 |
| **物化视图 / 投影** | 从事件派生出的读优化状态 |
| **数据框** | 面向分析的、支持批量变换的数据表抽象 |
| **矩阵 / 数组** | 面向数值运算和机器学习的表示 |

### R6 — 核心主旨（命题）

**命题 ①：数据模型不是“存储格式”，而是你思考问题的方式**
- 我的复述：你选了哪种模型，就决定了你更容易看到哪些关系、隐藏哪些关系，以及系统未来怎么改最省力。
- 我的例子：同样是一份用户资料，文档模型会让你自然地把它当成“一个完整对象”；关系模型会逼你把它拆成可复用的实体；图模型会让你关注“谁和谁连着”。

**命题 ②：文档模型适合树状对象，关系模型适合关系复杂的结构**
- 我的复述：如果数据本来就是一棵树，而且经常整棵读出来，文档会省掉很多 join；但如果你要经常引用任意节点、做双向查询，关系模型更稳。
- 我的例子：简历、订单、帖子详情页更像文档；组织、用户、地域、产品这种共享实体更适合规范化。

**命题 ③：规范化和反规范化不是好坏之争，而是读写与一致性的交换**
- 我的复述：你不是在“省不省事”之间选，而是在“写的时候麻烦一点”与“读的时候方便一点”之间选。
- 我的例子：Twitter/X 物化时间线只存 ID，不是因为它不懂反规范化，而是因为点赞数、用户名等字段变化太快，不值得把它们全塞进缓存里。

**命题 ④：图模型擅长的是路径，不是单点属性**
- 我的复述：图真正强的是“沿边走几步还能自然表达出来”，这件事在 SQL 里能做，但语法和执行都更别扭。
- 我的例子：查“出生在美国、现在住在欧洲的人”在 Cypher/SPARQL 里直观，在 SQL 递归 CTE 里就很长。

**命题 ⑤：GraphQL 的重点不是建模图，而是给前端定制数据形状**
- 我的复述：GraphQL 解决的是“客户端要刚好够用的数据”这个问题，不是“底层数据库应该是图”这个问题。
- 我的例子：聊天应用一次请求频道、最近消息、发送者、回复上下文，GraphQL 很合适，但底层可能仍然是关系库。

**命题 ⑥：事件溯源的核心价值是可重放、可演化、可审计**
- 我的复述：先把事实记下来，再按不同需求生成多个读模型，这样既能回放历史，又能换一种方式展示同一批事实。
- 我的例子：会议注册系统里，“预订已取消”比一堆字段变更更像业务语言，也更方便重建仪表板和座位分配视图。

**命题 ⑦：数据框与数组是把关系世界接到分析 / ML 世界的桥**
- 我的复述：当目标变成统计、向量运算、推荐模型时，表格不再是最终形态，而是要被整理成矩阵或稀疏数组。
- 我的例子：电影评分表转成“用户 × 电影”的矩阵后，才更容易做推荐算法。

### R7 — 论证链

**论证①：为什么文档模型常适合一对多树结构**
```
前提 1：很多应用天然以“一个对象 + 一堆子对象”的方式组织
前提 2：关系模型要把树切碎成多张表
前提 3：切碎会增加模式复杂度和应用层拼装成本
结论：→ 对于整棵一起读的树状对象，文档模型更自然
```

**论证②：规范化 / 反规范化的 trade-off**
```
前提 1：规范化只保留一份人类有意义的信息
前提 2：反规范化会复制信息到多个位置
前提 3：复制带来更快的读取，但更新和一致性更难
结论：→ 是否反规范化，取决于读写比例、变化频率和一致性要求
```

**论证③：图查询为什么比 SQL 更自然**
```
前提 1：图查询常需要变长路径遍历
前提 2：SQL 通常预先知道要做哪些 join
前提 3：图里你往往不知道要走几跳才到目标
结论：→ 图语言在“模式匹配 + 遍历”上更自然
```

**论证④：为什么 CQRS 有价值**
```
前提 1：复杂系统的写入语义和读取语义往往不同
前提 2：同一份表示很难同时把所有查询都做好
前提 3：事件日志可以作为稳定的权威源
结论：→ 把写和读分离，能同时获得可演化性和多种读优化视图
```

## 🔷 批判性阅读

### R9 — 我真的“懂了”吗？

- [x] 能用自己的话讲清楚关系、文档、图、事件流、数据框各自的定位
- [x] 能解释“规范化 vs 反规范化”的本质交换
- [x] 能说清 GraphQL 不是图数据库语言
- [x] 能把“事件日志 -> 物化视图”说成一个可执行思路

### 我的立场：大体同意，但有两个保留

**批判 1：图模型的强项有时被讲得过于理想化**  
类型：**(d) 分析不完整**

- 本章把图模型讲得很适合路径查询，但实际系统里，图数据库的运维复杂度、事务边界、索引策略和团队熟悉度也会显著影响选型。
- 也就是说，图模型“更自然”不等于“更适合上线”。

**批判 2：事件溯源的收益需要上下游工程配合才能兑现**  
类型：**(d) 分析不完整**

- 书里强调重放、审计、可演化，但没有把事件版本演化、幂等消费、外部副作用控制的工程成本展开到足够细。
- 如果团队没有成熟的事件治理能力，CQRS 很容易变成“多一层复杂度”而不是“多一层灵活性”。

**批判 3：模式灵活性并不免费**  
类型：**(c) 不合逻辑（对某些场景的隐含乐观）**

- 读时模式虽然降低了迁移成本，但它并没有消灭 schema 管理，只是把 schema 责任从数据库搬到了应用和测试上。
- 对大团队来说，这种转移未必是净收益。

## ✅ 行动清单

1. 下次设计表结构时，先问自己：数据更像**树**、**网**，还是**事实流**。
2. 如果准备做反规范化，先写清楚“哪些字段是派生数据、谁负责同步、失败时怎么恢复一致性”。
3. 如果查询里频繁出现“沿关系走几跳”，优先评估图模型或递归查询，而不是硬把 SQL 拉长。
4. 如果前端老是过取 / 少取，先考虑 GraphQL 或后端聚合层，但不要把它误当成图数据库。
5. 如果系统需要审计、重放、可重建读模型，再考虑事件溯源 / CQRS。
6. 如果目标是统计分析或机器学习，尽早把“关系表”思维切到“数据框 / 矩阵”思维。

## 一句话总结

**第3章的本质不是“比较数据库产品”，而是教你根据数据关系的形状来选择模型，并接受每种模型都会把复杂度转移到不同地方。**
