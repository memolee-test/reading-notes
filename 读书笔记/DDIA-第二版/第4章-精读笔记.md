# 第4章 精读笔记:存储与检索

> 基于 Adler《如何阅读一本书》分析阅读法,按"结构性 → 诠释性 → 批判性"整理。

## 🔍 检视报告

**归类:** 实用-原理类(Practical-Principle)。不是某个数据库的调优手册,而是教你「为什么这些存储设计存在」的原理框架,以便选型、调参、读文档时不蒙。

**一句话说清大意:** 这章想说明:数据库做两件事 —— 存进去 + 拿回来;OLTP 和 OLAP 各自演化出截然不同的存储引擎家族(日志结构 LSM 与就地更新 B 树,行存与列存),而当查询超出主键范围(地理、文本、语义),你需要另一套维度索引(R 树、倒排、HNSW)。

**作者要解决的核心问题:**
1. 最简单的键值数据库长什么样?它为什么不够用?
2. **日志结构 (LSM)** 与 **就地更新 (B 树)** 各自的设计哲学、读写性能、空间占用、崩溃恢复方案有什么差异?
3. OLTP 优化的存储引擎为什么放到分析负载就崩了?**列式存储**靠什么逆转这个局面?
4. 当主键和单列索引不够用时,**多维索引、全文检索、向量嵌入** 分别怎么解?
5. 现代云数据仓库为什么把存储、计算、表格式、目录解耦?

## 🔷 结构性阅读

### R1 — 分类

**实用-原理类**(Practical-Principle)。

- **实用**:直接指导存储引擎选型、索引选型、参数调优
- **原理**:不是某款数据库的操作手册,而是建立"为什么 LSM、为什么列存、为什么向量索引必须近似"的思维框架

### R2 — X-ray(全章主旨)

> 在有限的磁盘 I/O / CPU / 内存预算下,存储引擎的所有设计都是同一种问题的不同答案 —— 「把最需要的数据用最便宜的方式拿出来」;OLTP 与 OLAP 沿不同方向演化出 LSM/B 树和行存/列存两种哲学,而当查询超出主键,多维索引、倒排索引、向量索引就是这种"便宜拿数据"原则在新维度上的延伸。

### R3 — 大纲(重构论证骨架)

**第一部分:OLTP 存储引擎 —— 写多、读热数据、按主键找**
- 最简日志数据库 → 暴露索引为什么必要
- 日志结构家族:memtable + SSTable + 压实 + 布隆过滤器(LSM-Tree)
- 就地更新家族:页 + WAL + 写时复制(B 树)
- LSM vs B 树的四个维度对比:读、随机写、写放大、空间
- 二级索引 / 聚簇索引 / 覆盖索引 / 堆文件
- 全内存数据库的真正优势(不在"不读磁盘",在"不编码")

**第二部分:OLAP 存储引擎 —— 扫海量行、只看几列、聚合**
- 云数据仓库 + 数据湖解耦四件套(查询引擎 / 存储格式 / 表格式 / 数据目录)
- 列式存储:位图编码、游程编码、Roaring bitmap、排序顺序
- OLAP 的写入也是 LSM 风格(行内存缓冲 → 批量 flush 成列文件)
- 查询执行:JIT 编译 vs 向量化(殊途同归地利用 CPU)
- 物化视图与 OLAP cube(只为已知热查询服务)

**第三部分:超主键索引 —— 多维度、文本、语义**
- 多维索引(R 树 / Bkd 树 / 空间填充曲线)
- 全文检索 == 多维查询的特例(倒排索引 / n-gram / 莱文斯坦自动机)
- 向量嵌入(Flat / IVF / HNSW)—— 用准确率换速度的近似搜索

## 🔷 诠释性阅读

### R5 — 关键字与精确词义

#### OLTP 存储核心术语

| 术语 | Kleppmann 的精确含义 |
|---|---|
| **日志 (log)** | 磁盘上仅追加的记录序列,可能是二进制,不一定可读 |
| **索引 (index)** | 从主数据派生的额外结构,加快读取减慢写入 |
| **SSTable** | 按键排序、每键只出现一次的不可变文件格式 |
| **内存表 (memtable)** | 内存中的有序数据结构(红黑树 / 跳表 / 字典树),写入先入此处 |
| **段 (segment)** | 一个 SSTable 文件,写完即不可变 |
| **压实 (compaction)** | 后台合并段文件、丢弃过时值的过程 |
| **墓碑 (tombstone)** | 表示"已删除"的特殊记录,合并后真正消除旧值 |
| **LSM-Tree** | Log-Structured Merge Tree。基于 memtable + SSTable + 压实的引擎理念 |
| **布隆过滤器 (Bloom filter)** | 概率性"键不在此 SSTable"的快速检测器 |
| **假阳性 (false positive)** | 布隆过滤器说"有"但实际没有的情况 |
| **B 树** | 把数据库切成固定大小页、就地覆盖的索引结构,1970 年发明 |
| **页 (page)** | B 树的基本存储单元,4-16 KiB |
| **分支因子 (branching factor)** | B 树一页对子页的引用数,通常几百 |
| **撕裂页 (torn page)** | 部分写入导致的页损坏 |
| **预写日志 (WAL)** | Write-Ahead Log。修改 B 树之前先追加的日志,用于崩溃恢复 |
| **写时复制 (CoW)** | 修改时写到新位置而不是覆盖原页 |
| **写放大 (write amplification)** | 应用一次写入 = 底层 N 次写入的倍数 |
| **聚簇索引 (clustered index)** | 索引里直接存了实际数据 |
| **堆文件 (heap file)** | 行无序存储的文件,索引指向其中的位置 |
| **覆盖索引 (covering index)** | 索引里冗余了某些列,部分查询只看索引就够 |

#### OLAP 存储核心术语

| 术语 | 精确含义 |
|---|---|
| **行存 / 列存** | 一行连续存 vs 一列连续存 |
| **位图编码** | 把 n 个不同值的列变成 n 个位图,每行一位 |
| **游程编码 (RLE)** | 连续相同值的压缩(记录"几个连续 0 几个连续 1") |
| **Roaring bitmap** | 在稀疏 / 稠密两种位图表示间自动切换的紧凑结构 |
| **物化视图** | 查询结果实际写到磁盘的副本(虚拟视图是即时展开) |
| **数据立方体 (OLAP cube)** | 多维度预聚合的网格 |
| **JIT 查询编译** | 把 SQL 编译成机器码再执行,通常用 LLVM |
| **向量化处理** | 批处理列里的多个值(注意:这里"向量"≠ 嵌入向量) |
| **存储格式** | 文件级:Parquet / ORC / Lance / Nimble |
| **表格式** | 哪些文件组成一张表:Iceberg / Delta |
| **数据目录** | 哪些表组成数据库:Polaris / Unity Catalog |

#### 超主键索引核心术语

| 术语 | 精确含义 |
|---|---|
| **联合索引** | 多列拼接为单一键的 B 树索引(顺序敏感) |
| **多维索引** | 同时按多个维度查询:R 树 / Bkd 树 / 空间填充曲线 |
| **倒排索引** | 词项 → 包含该词项的文档 ID 列表 |
| **倒排列表 (postings list)** | 某个词项对应的文档 ID 集合 |
| **n-gram** | 长度 n 的子字符串(`"hello"` 的 3-gram: `hel`、`ell`、`llo`) |
| **莱文斯坦自动机** | 支持编辑距离搜索的有限状态机,Lucene 用它做模糊搜索 |
| **向量嵌入** | 浮点向量表示文本/图像/音频的语义位置 |
| **平面索引 (Flat)** | 全量比对的精确向量索引,O(N) |
| **IVF 索引** | 用质心做空间分区的近似最近邻索引 |
| **HNSW** | 多层小世界图的近似最近邻索引 |
| **余弦相似度 / 欧几里得距离** | 度量两个向量之间的"近" |

### R6 — 核心主旨(命题)

> **R6 验证条件**:每个命题都要能用自己的话复述 + 举一个能落到工程里的例子。

**命题 ①:索引是 trade-off,不是免费午餐**

- 我的复述:加索引让读快,但同时让写慢、让磁盘大;所以数据库不会默认给你所有列加索引,要你按查询模式手动选。
- 我的例子:在 Postgres 一张订单表上,如果你为 `user_id`、`status`、`created_at`、`product_id` 都加 B 树索引,一次 INSERT 会被放大成 5 次写(主键 + 4 索引),写吞吐直接腰斩。

**命题 ②:LSM 与 B 树是同一目标的两种哲学**

- 我的复述:**LSM 信"我不就地改,我追加 + 后台合并";B 树信"我就地改,但用 WAL 兜底"**。不存在谁全面赢,选择看工作负载和实现。
- 我的例子:RocksDB 在 Kafka offset、CockroachDB、TiDB 底层都是 LSM;PostgreSQL、MySQL InnoDB、SQLite 用 B 树。前者写吞吐能上百万 ops,后者范围查询和事务实现更稳。

**命题 ③:SSD 时代,顺序 vs 随机的差距变小但还在**

- 我的复述:HDD 上随机写是机械寻道,慢几个数量级;SSD 没机械,但"页可写、块才能擦 + GC"还是让顺序写更友好。
- 我的例子:在同一块 NVMe 上跑 RocksDB 和 LMDB 各自的 fio 基准,LSM 的写放大可能是 5,B 树是 20-30;不仅延迟差,SSD 寿命差距更大。

**命题 ④:OLTP 存储拿去跑 OLAP 必死**

- 我的复述:OLAP 一次扫万亿行只看 4-5 列,行存把整行 100 多列全拉进来,99% 的 I/O 是浪费的。
- 我的例子:在 MySQL 上跑 `SELECT SUM(quantity) FROM sales WHERE date>='2024-01-01' GROUP BY product`,跑几十分钟;同样数据导进 DuckDB,同样查询几秒。

**命题 ⑤:列式存储 + 压缩 + 排序是组合拳**

- 我的复述:列存让你**少读列**,压缩让你**少读字节**,排序让压缩**更狠**;三者互相增强,不能拆开评估。
- 我的例子:Parquet 文件里,如果你按 `date_key` 排序,date 列可能压到原来的 1%(因为日期重复值连续);后续的 product_sk 列按主排序键之内分组也会获得长重复段。

**命题 ⑥:数据仓库正在解耦成四层**

- 我的复述:从前一个 Hive / Snowflake 是一坨,现在拆成"查询引擎 + 存储格式 + 表格式 + 数据目录"四件套,让计算和存储能独立伸缩、让多引擎共享同一份数据。
- 我的例子:你用 Trino 查、用 Spark ETL,但底层都是 S3 上的 Parquet,顶上同一个 Iceberg 表格式,同一个 Unity Catalog —— 这就是当下数据湖仓的标准栈。

**命题 ⑦:OLAP 写入也是 LSM 风格**

- 我的复述:列存里"中间插一行"代价巨大,所以现代列存几乎都是:行存内存表 → 批量 flush 成列文件 → 后台合并。本质和 OLTP 的 LSM 同构。
- 我的例子:Snowflake、ClickHouse、Druid、Pinot 全是这套模式;只是 OLTP 的 LSM 段是按主键排序,OLAP 的列存段是按主排序列排序。

**命题 ⑧:全文检索 == 多维查询的特例**

- 我的复述:每个可能出现的词项是一个维度,文档要么有(1)要么没(0)。"红苹果"的搜索就是"红"维度和"苹果"维度同时为 1 的查询。
- 我的例子:在 Postgres GIN 索引和 Elasticsearch 的倒排索引底层,都是把"找包含 X 和 Y 的文档"翻译成两个位图的按位 AND —— 和列存数据仓库的"找 product='banana' AND store=3"是同一种操作。

**命题 ⑨:向量索引必须近似才能用**

- 我的复述:语义搜索的向量常常 1000+ 维,精确最近邻在高维度没有 sublinear 算法(维度灾难);只能用空间结构(质心 / 多层图)+ 贪心导航,把召回率从 100% 降到 95-99% 换速度。
- 我的例子:用 pgvector 在 1 亿条文档向量上,Flat 扫描几十秒,HNSW 几毫秒;代价是召回率约 95-98%,业务上能接受。

### R7 — 论证链

**论证①:为什么 LSM 写放大通常比 B 树低**
```
前提 1:B 树每次写必须写整页(4-16 KiB),即使只改了几个字节
前提 2:B 树还要写 WAL,所以同一条数据至少写 2 次
前提 3:LSM 的段是大块连续写出,SSTable 内部可以压缩
前提 4:LSM 也要写 WAL(memtable 持久化日志),但只在崩溃恢复时用
结论:→ 在覆盖密集型工作负载下,LSM 通常写放大更低
   → 写吞吐量更高 + SSD 寿命更长
```

**论证②:为什么列式存储 + 排序 + 压缩是组合拳**
```
前提 1:OLAP 查询通常只触发少数列(每次几列出来,不是 SELECT *)
前提 2:列内的值往往大量重复(产品类别、日期、店面 ID)
前提 3:位图 / RLE / Roaring 对重复值压缩效果极好
前提 4:沿主排序键排序后,主排序列会有更长的相同值连续段
结论:→ 列存让你少读、压缩让你更少 I/O、排序让压缩更狠
   → 三个杠杆相乘,而不是独立加成
```

**论证③:为什么数据立方体不能取代原始数据**
```
前提 1:cube 预计算的是固定维度组合的聚合
前提 2:新查询如果不在 cube 维度里(如"价格 > 100 美元的占比")无法回答
前提 3:cube 维度越多,组合数爆炸越严重(空间 / 重建成本)
结论:→ cube 只是"为已知热查询加速"的物化视图,不是 OLAP 主存储
   → 仍需保留原始数据来回答未知的查询
```

**论证④:为什么向量索引必须近似**
```
前提 1:语义搜索的向量常常 1000+ 维
前提 2:精确最近邻在高维下没有 sublinear 算法(维度灾难)
前提 3:Flat index 是 O(N),N 上亿就不可接受
前提 4:IVF / HNSW 用空间结构 + 贪心导航,把搜索压到亚线性
结论:→ 在向量空间维度灾难下,工程上唯一可行的是用准确率换速度
   → 召回率 95-99% 是默认现实
```

**论证⑤:为什么"嵌入式数据库"在重新流行**
```
前提 1:很多场景数据量不大,完全可以单机
前提 2:进程内调用比网络 RTT 快几个数量级
前提 3:多租户场景里,每租户一个独立嵌入式 DB 反而隔离更干净
前提 4:RocksDB / SQLite / DuckDB / LMDB 已经把嵌入式做到生产级
结论:→ 在数据量适配的场景下,嵌入式数据库的性能优势远大于服务器化数据库
```

### 我的立场:大体同意,但有 5 个保留

**批判 1:LSM vs B 树的比较过于"经验法则化"**
类型:**(d) 分析不完整**

- 书里给出"LSM 写多更好、B 树读多更好"的口诀,但工业现实里:
  - LSM 的**长尾延迟**(背压 / 压实风暴)只用一句话带过,没展开
  - B 树+并发(锁 / MVCC / 页分裂阻塞)对真实写吞吐的影响没讨论
  - "写多 / 读多"的边界在哪里?70/30?90/10?完全没刻度
- 这种 binary 二选一在真实选型时是误导性的 —— 实际看你能否容忍长尾、是否有强一致需求、引擎成熟度、运维熟悉度。

**批判 2:写放大的讨论缺少"数值锚点"**
类型:**(d) 分析不完整**

- 书里反复说"LSM 通常写放大更低",但没给典型数字:
  - B 树:~10-30(算上 WAL + 整页写 + 页分裂)
  - LSM (level compaction):~3-10
  - LSM (WiscKey 风格 KV 分离):~1-3
- 没有这些量级,读者无法做工程决策(到底买多大的 SSD、SSD 几年内会写废)。

**批判 3:RUM 猜想引用了但没讲透**
类型:**(d) 分析不完整**

- RUM 猜想:Read overhead / Update overhead / Memory overhead 这三个目标,**你只能同时优化两个**(参考文献 27)。
- 这是理解 LSM、B 树、Bw-tree、FractalDB 等所有存储引擎设计的根本统一框架。
- 书里只作为脚注引用,**没有把它提升为统领全章的视角** —— 错过了一个能让整章"立起来"的工具。

**批判 4:云数据仓库的经济模型完全没讲**
类型:**(c) 不合逻辑**(对企业读者的关键遗漏)

- BigQuery、Snowflake 在工程上的颠覆性是**架构解耦**,但在商业上的颠覆性是**计费模型**(BigQuery 按扫描字节、Snowflake 按 warehouse 时间)。
- 这种计费模型反过来**直接驱动了"为什么列存、为什么 Parquet、为什么分区、为什么 cluster key"** 等工程决策。
- 只讲架构不讲经济 → 工程师读完仍然不知道**什么时候该跳上云仓**(数据量?查询频率?预算曲线?)。

**批判 5:向量索引的"近似 trade-off"讲得太干净**
类型:**(d) 分析不完整**

- 书里讲了 Flat / IVF / HNSW 三种,但生产里的真正痛点没讲:
  - **重新嵌入(re-embedding)的成本** —— 换嵌入模型 = 整个索引重建
  - **向量索引的写入 / 更新是难题** —— HNSW 删除节点是出名的麻烦
  - **召回率 90% 与 99% 的延迟差几个数量级**,业务该选哪档没刻度
  - **IVF 的 partition 漂移**(数据分布变化后,质心可能不再代表)
- 入门读者会以为"上 HNSW 就行了",其实生产里还有不少坑。

## ✅ 行动清单

1. **选 OLTP 存储引擎时**,先量化写读比例和长尾延迟容忍度 —— 决定 LSM 还是 B 树倾向,不要只看"写多读少"这一维。
2. **调 RocksDB / Cassandra** 时,先看压实策略(Size-tiered vs Leveled)和背压设置;**写吞吐量瓶颈往往不在写,在压实跟不上**。
3. **设计 OLTP 表的二级索引**时,默认信念:每加一个索引 = 写放大上升、表更大、可能触发更多 WAL。**只为最高频查询建,不要"以备万一"**。
4. **涉及分析查询**时,首选列式存储(单机 DuckDB / 共享 Parquet on S3 / 云上 Snowflake / BigQuery),**不要在 OLTP 库上硬撑**。
5. **设计列存表**时,把"最常出现在 WHERE 里的列"作为主排序键(Snowflake 的 cluster key、Iceberg 的 sort order),压缩效果会被你拉满。
6. **用数据立方体 / 物化视图**只当作"已知热查询的预聚合",**不要当主存储**;保留原始事实表回答未知查询。
7. **做全文检索**时,先评估 PostgreSQL GIN 是否够 —— 数据量在百万-千万级且查询不复杂时,**先别上 Elasticsearch**(运维成本远高于多数人估计)。
8. **做向量检索**时,先用 pgvector / Faiss HNSW;**真到了亿级 + 召回率 95%+ 时再考虑专用向量数据库**(Pinecone / Weaviate / Milvus / Qdrant)。
9. **写 WAL 时记得 fsync()** —— 不 fsync 等于没写,崩溃就丢。同时关心 OS page cache 行为(直 fsync 还是 flush + fdatasync)。
10. **设计备份策略**时,**LSM 的快照天然便宜**(记录段文件清单即可),**B 树要做 base + incremental WAL**;选数据库时把这件事算进去。
11. **遇到"为什么 OLTP 数据库跑分析慢"的提问时**,直接拿出本章的"行存 vs 列存"那张图给同事看,**不要花一小时调 SQL,先换存储**。
12. **理解一个数据库时**,问三个问题:(1) 它是 LSM 还是 B 树?(2) 它是行存还是列存?(3) 它的二级索引是聚簇 / 堆 / 覆盖?三个答案合起来就是这个数据库的"DNA"。

## 一句话总结

**第 4 章的本质不是"介绍数据结构",而是教你看清楚:所有存储引擎的设计都是同一个问题的不同答案 —— 在磁盘 I/O / CPU / 内存 / 寿命这几条预算下,把最需要的数据用最便宜的方式拿出来;LSM 和 B 树是 OLTP 的两条路,行存和列存是 OLTP 与 OLAP 的分水岭,而当查询超出主键(地理、文本、语义),你只是把同一个"便宜拿数据"的原则推到新维度而已。**
